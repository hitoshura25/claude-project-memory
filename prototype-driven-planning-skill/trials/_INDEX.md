> Add new tags here as they emerge. Keep tags lowercase, hyphenated.

| Tag | Meaning |
|-----|---------|
| `clean-sweep` | All tasks/phases passed |
| `scaffold-bug` | Scaffold task ran from wrong directory or missing config |
| `bootstrap-gap` | Tooling environment not available after scaffold |
| `tdd-violation` | TDD pairing constraint violated |
| `schema-validation` | PydanticAI schema caught an error |
| `circuit-breaker` | Task hit retry limit and escalated or stopped |
| `aider-scripting` | Issue with Aider CLI flags or message file |
| `context-exhaustion` | Model ran out of context / OOM / repetition loop |
| `prototype-reference` | Issue with prototype inlining or reference |
| `cwd-selection` | Wrong working directory for task execution |
| `path-rebasing` | File paths not correctly rebased for service root |
| `rate-limit` | Cloud executor hit usage rate limit |
| `enum-serialization` | Pydantic enum not serialized to string |
| `task-sizing` | Task too complex for target model capacity |
| `stub-gap` | Test task imports module with no stub; causes ImportError instead of NotImplementedError |
| `cross-task-drift` | Field names, signatures, or contracts mismatch across tasks due to no shared validation |
| `lint-loop` | Model burns reflection cycles on auto-fixable lint errors |
| `config-drift` | Config value differs between skill reference (desired) and generated pipeline (actual) due to regeneration from model memory |
| `lint-format-gap` | Linter auto-fix + formatter can't fix a lint error class (e.g., E501 on long string literals) |
| `stub-pass` / `stubs-pass-defensive` | Test passes against a stub instead of raising stub error, because test uses defensive defaults or asserts on placeholder values |
| `stub-mock-import` | Stub missing import of third-party dependency that tests mock at module boundary |
| `split-module-tests` | One module split across multiple tasks, all sharing a single test file; test gate runs whole file so no individual task can pass |
| `mock-path-inconsistency` | Same test file patches the same dependency via multiple different module paths |
| `test-writing-coherence` | Test-writing model produces individually-correct tests that collectively drift |
| `partial-stub-gap` | Component has mixed declarative + stubbed parts; `verify_task`'s single-pattern check can't model this |
| `verify-rigidity` | Pipeline's verification check rejects correct output due to structural gap in what the check can recognize |
| `test-over-specification` | Tests assert on details absent from task spec, blocking any implementation from satisfying them |
| `fixture-path-bug` | Test fixture uses path-resolution logic that resolves wrong under the current project layout |
| `system-prompt-bloat-fixed` | Tightening the per-prompt system-message content eliminated a failure class |
| `lossy-prose-transport` | Task description prose contains a specific command, contract, or value that the downstream skill has no schema field to read; information is lost in translation |
| `test-command-gap` | Per-task verification command lives only in prose; generator substitutes a generic default that doesn't meet the task's actual needs |
| `integration-test-no-services` | Integration test runs without the external services it depends on being started; fixtures fail by design |
| `silent-scope-removal` | Model drops an approved-scope item mid-phase without surfacing the removal to the user (often framed as "for the prototype, X isn't needed") |
| `table-as-complete-spec` | Model reads a reference table in the skill as an exhaustive specification (one cell per question) rather than as a starting point for reasoning about additional surfaces or options |
| `prompt-consumer-pipeline` | Pipeline reads pre-generated prompts from the prompt-composition skill verbatim instead of composing prompts at runtime; single source of truth for prompt content |
| `refactor-architecture-validated` | Trial confirms a refactor's structural promises hold end-to-end under load (e.g., I01's no-mutation invariant, new log paths, precondition checks) even when the trial ultimately fails for unrelated reasons |
| `tier-1-timeout-too-tight` | Pipeline timeout sized for tier-0 executor (Aider) is too short for tier-1 executor (Claude CLI) on larger prompts; manifests as silent timeouts with no diagnostic output |
| `executor-output-vs-code-quality-conflation` | Pipeline's verification step (lint/test) reports executor-output failures (file not created) as if they were code-quality failures (E902 No such file or directory treated as a lint error) |
| `gemini-tool-calling-bug` | Gemini CLI emits tool calls with prompt body fragments as path arguments, producing ENAMETOOLONG errors and partial/missing output |
| `timeout-no-diagnostic-output` | When `subprocess.run` is killed at timeout, only "timed out after Ns" is captured; partial stdout/stderr is lost, making it impossible to tell whether the executor was making progress or stuck |
| `surface-coverage-gap` | Security tooling runs only against the primary language surface (e.g., bandit on Python) and misses additional file surfaces carried by the prototype (Dockerfile, docker-compose, IaC, shell) |
| `judgment-as-fact` | Model states a judgment call in observation-shaped prose, without citing evidence or labeling it as inference — makes the claim indistinguishable from prototype observation to a future reader |
| `feasibility-in-disguise` | An open question framed as an operational decision (passes the "difference test" of not changing architecture) but requires the design doc to assert something the prototype didn't observe (fails the "assertion test") |
| `deferred-decisions-abuse` | Items placed in Deferred Decisions that should have required user decision or prototype extension, masked by "implementation-phase decision" framing |
| `phase-1-scope-ambiguity` | Design doc conflates user-approved Phase 1 scope deferrals with model-designed prototype minimum-viable-validation limitations; downstream readers can't distinguish the two |
| `severity-blind-handling` | Security findings treated uniformly regardless of Critical/High/Medium/Low severity; routine deferral path offered for Critical findings that should block |
| `environmental-assessment-shortcut` | Model uses contextual CVSS reasoning ("this doesn't apply to us because we're not running a public proxy") to accept findings without first exploring mitigations; environmental assessment finalized unilaterally without user review |
| `transitive-reachability-handwave` | Assessment that vulnerable code is unreachable lacks specific evidence (module-level import graph, reachability analysis) — relies on shape-of-argument rather than concrete observation |
| `lesser-evil-tradeoff` | Model presents two options and picks the less-bad one without exploring the fuller option space (pinning, downgrading, overriding, excluding, replacing) |
| `cross-component-misplacement` | A security scenario is attached to the file of a component that consumes the result of an action rather than the file of the component whose code performs the action; valid format and real concern, but wrong placement |
| `id-set-drift-class` | Set of OWASP IDs in `components.yml` and the set cited in per-file scenario headings can drift independently because the validator only format-checks IDs, never compares sets |
| `validator-gap` | A failure class is mechanically preventable (sets must match, fields must be present) but the validator doesn't check it; closing the gap is a structural fix, not a model-behavior fix |
| `silent-actor-assumption` | A security scenario assumes the file's component is the actor; reviewer cannot tell whether the model considered other components without an explicit Performed-by field |
| `stale-spec-pin` | Skill reference doc pins to an outdated version of an external standard (e.g., OWASP ASVS) without verifying current state; the pin reads as authoritative but no verification was done |
| `pseudo-verification` | Prose in a skill or doc claims verification that didn't happen (e.g., "as of <date>"); the parenthetical reads as evidence of a check, but the check was never performed |
| `triple-source-of-truth` | Same canonical fact (e.g., a category label) stored across three artifact types (reference doc prose, per-scenario field, dedicated data file) — three drift surfaces for one fact |
| `version-baked-id-missing` | Standard's older ID format omits version (e.g., ASVS 4.x's `V<n>.<n>.<n>`); citations across spec versions become ambiguous and version-mismatch is invisible at the ID level |
| `validation-run` | Run that exercises a previously-built skill against a real feature with no skill changes intended; success means no new failure modes surface |
| `stable-id-citation` | Downstream skill cites upstream content via stable, validator-enforced identifier (kebab-case slug, unique within scope) instead of by name or by prose paraphrase; eliminates the leaky abstraction where regeneration silently breaks downstream references |
| `schema-extension` | Skill change adds new schema fields and validators without breaking the existing schema's invariants — additive change ships as a new field with its own validators, leaves prior fields untouched |
| `roadmap-integration` | Decomposition skill consumes roadmap output (components.json + roadmap.json) as primary input; component boundaries and behavioral scenarios come from the upstream typed artifact, not from re-inferring against the design doc |
| `cwd-relative-path-resolution` | Path resolution code uses `Path(p).resolve()` (CWD-implicit) when the contract requires resolving against an explicit project root; works in some calling contexts (validation from project root) and silently fails in others (pipeline runtime from a subdirectory). Structural fix is a named helper that takes its anchor explicitly. |
| `host-runtime-mismatch` | Bootstrap command depends on the host's default runtime version (e.g., `python3`, `node`) instead of the project's pinned runtime version. Works when the host's default happens to match the pin; fails silently when it doesn't. Structural fix is to make the runtime version pin explicit and have bootstrap request it by name. |
| `bootstrap-cascade` | Bootstrap failure inside the pipeline triggers the standard retry/escalation/cascade logic, which is shaped for code-quality failures and wastes retries on environmental issues. Downstream tasks are skipped on a failure class no executor can recover from. |
| `imported-config-immutable` | Long-running process imports config at startup; mid-run on-disk edits to that config don't take effect because the imported values are already in memory. An executor that diagnoses a config issue and writes a fix produces a misleading "fix landed" report when the runtime never sees the change. |
| `environmental-vs-code-failure-recovery-mismatch` | A uniform recovery loop (retry-with-same-input → escalate-to-stronger-executor → cascade-skip) is applied to a failure class whose recovery requires different intervention (interactive diagnosis with the user, environmental fix, configuration regeneration). The structural fix is to detect the failure class earlier and route it to the appropriate recovery path — often by removing the class from the loop's scope altogether. |
| `static-table-as-stop-thinking` | Skill reference contains a static "for ecosystem X, use tool Y" table; the model running the skill reads it as authoritative, picks the row, and skips verification. Recurring shape across CLI flags, OWASP version pins, and bootstrap commands. The structural fix is to specify the *responsibility* (what the output must accomplish) and *signals* (what evidence to look for in the project) and have the model research the *realization* at use time. |
| `cliff-edge-discipline` | A planning-skill discipline of labeling design-doc observations that document specific failure modes for an alternative configuration. The label `**Cliff edge:**` requires both sides: the working pattern *and* the failure mode for the alternative. Downstream skills (roadmap, decomposition, prompt-composition) pick this up as a structural signal for `criticality: cliff_edge`. |
| `breadcrumb-elicitation` | Phase-2 (prototype) iteration discipline of capturing "tried X, got error E, switched to Y" failure breadcrumbs in the prototype README's Toolchain notes, *plus* deliberate probing for boundaries that incidental iteration doesn't surface. Without breadcrumbs, the design doc has no source material for cliff-edge labels. |
| `labeling-as-signal` | The discipline of using a typed prose label (`**Cliff edge:**`, `**Not observed:**`, `**Prescribed (not validated):**`) as a structural signal that a downstream validator can detect via regex/count and enforce coverage on. Not just documentation — a mechanical pickup point that closes a gap where models would otherwise infer criticality from prose inconsistently. |
| `structural-signal-from-prose-label` | The pattern of converting an inline prose convention into a structural signal: a regex-detectable label in one skill becomes a required typed field in the next skill's schema, with a cross-file validator ensuring count parity. P04 → R04 → D03 → PC01 chain demonstrates the pattern end-to-end. |
| `upstream-vs-downstream-fix` | When a failure surfaces downstream (e.g., executor outputs the wrong file), the temptation is to fix the downstream skill. The upstream fix — making the failure structurally impossible by adding a typed signal upstream — is often more durable. The arc's signature pattern: a failure at the executor layer routed back through the prototype → design-doc → roadmap → decomposition → prompt chain, with the fix landing at the upstream layer. |
| `criticality-tier` | A three-tier classification of behaviors in roadmap scenarios: `cliff_edge` (non-negotiable boundary with documented failure mode for the alternative), `behavioral` (proven path the prototype demonstrated), `prescribed` (design-doc prescription not present in the prototype). Drives how downstream skills frame each scenario in executor prompts. Independent of `evidence_kind`: a prototype-evidence scenario may be cliff-edge or behavioral; a prescribed scenario typically carries `prescribed` criticality but may carry `cliff_edge` if the prescription describes a boundary with a specific failure mode. |
| `verification-form` | A four-value hint to the test-writing executor about what form a scenario's test takes: `tool_invocation` (run a tool, assert on exit/output), `unit_test` (pytest function with mocks), `integration_test` (pytest function with real services), `static_assertion` (grep/regex/AST check). Not a strict contract; some scenarios have two reasonable forms. The field's job is to give the prompt-composition skill a consistent framing signal. |
| `cliff-edge-coverage-validator` | A validator pattern that counts cliff-edge signal occurrences in an upstream artifact and requires ≥ that many cliff-edge instances in a downstream artifact. R04's check: `**Cliff edge:**` labels in design doc ≤ `criticality: cliff_edge` scenarios in roadmap.json. D03's parallel check: roadmap cliff-edge scenarios ≤ task citations across `roadmap_functional_scenarios` and `roadmap_security_scenarios`. Lower-bound coverage, mechanical via count comparison, fails with named missing IDs. |
| `cliff-edge-coverage-at-task-layer` | D03's specific application of the cliff-edge coverage pattern: every cliff-edge scenario in roadmap.json must be cited by at least one task. Forced the relaxation of the original "scaffold/infra tasks may cite none" rule to let scaffold tasks cite tool-invocation scenarios their `test_command` verifies. |
| `prototype-decomposer-only` | The decomposition-skill discipline that the prototype is an input to the decomposer's planning phase but not referenced in task output. No prototype paths in task descriptions, no inlined snippets, no "see prototype X" prose. Chosen over path-only references (which would give executors a path they cannot use, since the implementation pipeline doesn't surface the prototype to executors). |
| `relaxed-scaffold-citation` | The D03 rule change that lets scaffold/infrastructure tasks cite tool-invocation scenarios their `test_command` verifies, instead of leaving citation lists empty. The original rule was an under-specification; the relaxation makes citation uniform across task types and lets the cliff-edge coverage validator work without special cases. |
| `schema-field-removal` | The pattern of dropping a schema field that has become a contradiction surface (e.g., `FileChange.description`) rather than rewriting prose conventions to disambiguate it. The field's job is taken over by structured slots in the surrounding artifacts (task-level description, cited scenarios, criticality framing); removing the redundant surface eliminates the failure mode at the source. |
| `structural-signal-chain` | A multi-layer pattern of typed signals flowing through skills, with each layer's validator enforcing coverage of the previous layer's signal. The cliff-edge arc instantiated this: prototype breadcrumb → design-doc label → roadmap criticality → task citation → prompt subsection framing, with validators at each transition. No layer can silently drop a cliff edge. |
| `task-type-specialized-preamble` | The prompt-composition pattern of dispatching to one of two preamble files based on `task.task_type.value`. Test tasks get the test-writing role and stub-writing rules; implementation tasks get the impl-task role with cliff-edge non-negotiability framing and stub-replacement rules. Replaces a single universal preamble that had to share role description across both task types. |
| `criticality-grouped-behaviors` | The prompt-composition pattern of grouping cited scenarios by criticality (cliff_edge → behavioral → prescribed) instead of by kind (functional vs security). Subsection headers (("Cliff edges (non-negotiable boundaries...)", "Behavioral contract (proven path from prototype)", "Prescribed (security or production additions; not in prototype)")) carry instructional load. Functional/security distinction is preserved at the per-scenario render level via OWASP fields. |
| `verification-form-rendered` | The prompt-composition rendering of R04's `verification` field as a per-scenario `**Verification form**: <value>` line. Hints at the test form the executor should produce. Lives alongside `Verified by` (the specific verifier) and per-scenario fields like Given/When/Then. |
| `description-field-removed` | The state where `FileChange.description` is no longer in the decomposition schema (dropped in D03) and consequently no longer rendered in the prompt-composition Files section (updated in PC01). Files entries carry only path, operation, and stub flag (the latter as a `; stub` suffix when set). |
| `structural-signal-rendering` | The prompt-composition pattern of rendering typed upstream signals (criticality field as subsection header; verification field as per-scenario line; stub flag as file-suffix; absent description field) into the executor prompt without re-introducing prose. The terminal layer of a structural-signal-chain. |
| `single-source-of-truth-for-prompt-rendering` | The implementation-skill pattern of consuming pre-generated prompt files verbatim instead of re-rendering them at runtime. The prompt-composition skill is the sole producer; downstream consumers (the pipeline, manual users, debug inspections) all see the same file content. Eliminates a class of drift where the pipeline's runtime rendering would lag behind the upstream skill's evolution. |
| `prompt-mutation-eliminated` | The state where the pipeline cannot modify prompt content between retry attempts. Retry/escalation context flows through the prompt's pre-rendered `## Previous Attempts` section pointing at on-disk error-log paths, not through prompt re-rendering. Logged byte counts confirming identical prompts across attempts are the structural witness. |
| `co-located-execution-artifacts` | The convention of placing per-attempt error logs at `tasks/<feature>/logs/<lint\|tests>/<task-id>/attempt-<N>-tier<M>.txt` — sibling to the `tasks/<feature>/prompts/` directory the prompts live in. Project-root-relative, named in the prompt with explicit "paths relative to the project root" framing, written by the orchestrator (any orchestrator). Replaces the older pipeline-relative `pipelines/<feature>/logs/` convention which had ambiguous path resolution when executors ran from non-pipeline CWDs. |
| `prerequisite-validation-at-startup` | The implementation-skill pattern of validating upstream-skill outputs at pipeline startup (in `check_preconditions`) rather than discovering missing inputs mid-run. The prompts directory and per-task `<task-id>.md` files are checked once, before the execution loop; missing files fail fast with a pointer back to the prompt-composition skill. The structural counterpart to schema-side validation in upstream skills. |
| `structural-signal-chain-completed` | The state where every layer in a multi-skill chain has typed signals enforced by validators, with no remaining surface where the signal could be silently stripped. The cliff-edge arc instantiated this: prototype breadcrumb (Phase 2 of planning) → design-doc label (P04) → roadmap criticality (R04) → task citation (D03) → prompt subsection framing (PC01) → executor consumption (I01). Each transition has a validator; the executor receives the structural signal intact. |
